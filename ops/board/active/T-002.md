## Why this exists
Two commands lie when run inside the repo that BUILDS POLARIS, and both lie quietly.

**`update` is a no-op that looks like a success.** It fetches `main`'s tarball and installs from
`<root>/ops/install.sh` — which, in this repo, is our own *installation*. So `update` here installs
`ops/` over `ops/`, prints "updated 5.6.0 → 5.6.0", and brings across exactly none of the kit code you
just wrote in `kit/`. The command a maintainer actually wants is `python kit/ops/pack.py --dogfood`,
which installs the *published release* — one letter of intent away, and nothing says so.

**`doctor` cannot see the one skew that matters here.** When `kit/ops/VERSION` is ahead of
`ops/VERSION`, this repo has built a release it has never run — and, because `main`'s tarball and the
raw channel URL both serve that installation, every kit installed in the world is still being handed
the old one. CI's daily job catches it; the human standing in the repo does not.

## Acceptance criteria (binary — each becomes a test; put the runnable ones in verify:)
- [ ] Self-hosting is detected exactly as `ops/contracts/self-hosting.md` specifies (the `pack.py` tell). One file test, no git, no network.
- [ ] In a self-hosting repo, `update` REFUSES rather than no-ops: it explains that `ops/` here is an installation, that updating it from `main` would install it over itself, and names `python kit/ops/pack.py --dogfood` as the command that was meant. Exit non-zero.
- [ ] `update` behaves exactly as it does today in every ordinary target repo — no new prompts, no new output. This is the common path; do not regress it.
- [ ] `doctor` in a self-hosting repo prints one extra line comparing `kit/ops/VERSION` to `ops/VERSION`: in sync → a quiet ✅; kit ahead → a ⚠ naming both versions and saying the release has not been dogfooded, with the `--dogfood` command.
- [ ] `doctor` in an ordinary repo prints nothing new.
- [ ] `doctor --selftest` still passes (it drills a throwaway repo, which is NOT self-hosting — a good regression check that detection does not misfire).
- [ ] Bash 3.2 compatible. No new dependencies.
